Executive Summary

This plan merges the best mechanisms from Cavekit, Superpowers, Matt Pocock's skills, and Soul Protocol into one agnostic, self-triggering skill architecture under your agent-skills repository.

The core idea: auto-triggered disciplines (the reusable substance) are invoked by user-triggered loops (entry points that produce a named deliverable). A thin adapter layer provides session bootstrap without coupling any skill to a specific harness.

Why this is different. Most skill systems are either auto-triggering but harness-coupled (Superpowers), or user-invoked and composable (Matt), or file-centric and token-lean (Cavekit). This design keeps auto-triggering as the default while making every user skill a named loop with a named deliverable — and keeps all skills agnostic of any specific agent, model, or tool.

Design Principles

1. Behavior triggers domain

Process loops (behavior) discover and invoke domain capabilities. Domain skills never invoke process skills.

2. Self-triggering by default

Most skills are auto. They pop up based on context, symptoms, or behavior. User invocation is the exception.

3. Loops produce deliverables

Every user-invoked skill must declare its loop and its deliverable. No generic "help me" entry points.

4. One plan per objective

Each fix, feature, spike, or refactor gets its own /docs/plans/<date>-<type>-<slug>/ folder.

5. Compression with clarity

The caveman skill compresses spec-adjacent prose without obscuring meaning. Symbols limited to → and §.

6. Agnostic at the core

Skills express behavior. The adapter maps behavior to tools. Only the adapter knows harness-specific names.

Triggering Model

Every skill declares invocation: in its frontmatter. There are three values, with strict contracts:

auto Default

Self-triggers on context, symptoms, or behavior. Disciplines and most domain capabilities.

Examples: tdd, debug, verify, use-clickup, caveman.

user Loop entry only

Human invokes a named loop that produces a specific deliverable. Must declare loop: and deliverable:.

Examples: /grill, /plan, /build, /gestionar-cursos.

bootstrap Session enabler

Injected at session start and after compaction. Only one skill holds this role.

Example: bootstrap (teaches auto-triggering itself).

Enforceable rule. skill-forge rejects a user-invoked skill that does not declare both loop: and deliverable:. This eliminates ambiguous entry points like Matt's grill-me vs Superpowers' brainstorming collision.

Memory vs Artifacts

A previous version of this plan conflated documentation artifacts with memory. They are different.

🧠 Agent Memory

What: Event remembrance — what happened, what you know, how to do things.

How: System prompt, Soul Protocol, or other memory substrate.

Scope: Cross-project, tied to the agent.

📋 Project Artifacts

What: Deliberately written documentation — requirements, designs, plans, lessons.

How: Markdown files in /docs/plans/.

Scope: Project-local, version-controlled.

The critical habit: the agent must remember to check and remember to update the project artifacts. That habit is taught by the bootstrap/system prompt and reinforced by memory. The artifacts themselves are not memory — they are the externalized state the agent maintains.

Soul Protocol is out of scope for this skills redesign. It is core infrastructure you already have. These skills must work whether Soul is present or not. If Soul reinforces the artifact-check habit, that is a bonus — not a requirement.

The write habit

When something important happens, the agent writes it to the appropriate artifact:

  • A new term is coined → update CONTEXT.md.
  • A requirement is clarified → update PRD.md in the active plan.
  • An invariant is discovered → update ARD.md or SPEC.md.
  • A bug is fixed → append to LESSONS.md.

Per-Plan Artifact Structure

Each objective (fix, feature, spike, refactor) gets its own plan folder. The folder contains the basic artifacts that comprise the development cycle for that objective.

repo-root/ ├── CONTEXT.md ← shared ubiquitous language └── docs/ └── plans/ ├── 2026-06-25-fix-auth-timeout/ │ ├── PRD.md │ ├── ARD.md │ ├── SPEC.md │ ├── PLAN.md │ └── LESSONS.md ├── 2026-06-26-feature-oauth-flow/ └── 2026-06-27-spike-realtime-sync/

Naming convention

/docs/plans/<yyyy-mm-dd>-<type>-<slug>/

  • type — fix, feature, spike, refactor, chore, docs, test
  • slug — short hyphenated description

Right-size within a plan

Artifacts are created lazily. The type signals intent, not rigidity.

Objective sizeArtifacts created
One-line fixPLAN.md only (+ LESSONS.md if bug)
Small featurePRD.md + PLAN.md
Medium featurePRD.md + ARD.md + PLAN.md
Complex / high-blast-radiusFull set: PRD + ARD + SPEC + PLAN + LESSONS

Plan selection

The agent determines the active plan by:

  1. User names it: /build docs/plans/2026-06-25-fix-auth-timeout
  2. Conversation context: the plan slug matches what the user is discussing.
  3. Most recently modified plan folder, if still ambiguous.
  4. Ask, if still ambiguous.

Archiving

When all PLAN.md tasks are x, optionally move the folder to /docs/plans/archive/ to keep the active workspace clean.

Ubiquitous Language — First-Class Concern

Borrowed from Matt Pocock's domain-modeling skill, ubiquitous language is not a side document. It is a cross-cutting discipline that every skill participates in.

Core rule. Every skill reads CONTEXT.md if it exists. Every name in code, spec, or plan must align with the glossary. Fuzzy or overloaded terms are sharpened. New terms are coined explicitly and written to CONTEXT.md immediately.

CONTEXT.md format

Each entry has a canonical term and a list of synonyms to avoid:

## materialization cascade

When a lesson inside a section of a course is made real (given a spot in the file system).

_Evitar_: "lesson becomes real", "section activation", "file system placement"

How skills use it

SkillHow it uses CONTEXT.md
grillCreates/updates glossary while sharpening requirements.
designUses canonical terms in ARD; challenges overloaded terms.
specUses canonical terms in SPEC.md.
planNames tasks using project language.
buildNames variables/functions/files from glossary.
reviewChecks that code and docs use terms consistently.
domain-modelingAuto-updates glossary when terminology is fuzzy.

The payoff: fewer tokens per conversation, less misalignment, and code that reads consistently across the codebase.

caveman — Compression Without Obscurity

Keeping the original name. The skill compresses spec-adjacent prose to save tokens while staying readable for a non-always-technical reader.

Compression toolkit (5 layers, in priority order)

Grammar is the biggest token win. Symbols add density. Use words for anything obscure.

1. Grammar

  • Drop articles (a, an, the).
  • Drop filler (just, really, basically, simply, actually).
  • Drop auxiliary verbs where fragments work (is, are, was).
  • Drop hedging (might, perhaps, could be worth).
  • Use short synonyms: fix > implement, run > execute.
  • Fragments are fine.

6. Semantic emoji (chat-only, fixed traffic-light palette)

Rule: emoji are a chat-only affordance for communicating to the human in the terminal/shell. Use a fixed traffic-light palette — three circles for severity, one glyph for in-progress. Nothing else.

NEVER in artifacts: no emoji in .md files, commit messages, code, or shell output. Emoji unicode is garbage in persisted artifacts. Artifacts use ASCII words or simple chars instead.

Chat emojiArtifact ASCIIMeans
🟢PASS / OK / ✓done / holds / ok
🟡WARN / HARDENwarning / should fix
🔴FAIL / BLOCK / ✗ / NEVERfailed / critical / forbidden
🔵INFOnote / informational
🚧WIPin progress

Forbidden (too many variants): 🟠 🟣 ⚫️ ⚪️ 🟤 and all square variants 🟥 🟧 🟨 🟩 🟦 🟪 ⬛️ ⬜️ 🟫. Also drop ✅ ❌ ⚠️ 🛑 from chat — redundant with the traffic-light circles. Keep ✓ / ✗ as artifact ASCII chars.

7. Directional (chat-only, magnitude & trend)

Directional emoji express change and trend — distinct from -> / => which express logical flow. Chat-only; artifacts use ASCII.

Chat emojiArtifact ASCIIMeansExample
⬆️+ / upincrease / upgrade⬆️ retry to 5
⬇️- / downdecrease / downgrade⬇️ batch to 50
↗️up trendtrend up (metrics, gradual)↗️ p95 latency
↘️down trendtrend down (metrics, gradual)↘️ error rate
↔️<->bidirectional / two-way↔️ sync both ways

Reserved for navigation/sort only (use sparingly, context-dependent): ◀️ ▶️ previous/next, 🔼 🔽 priority/sort up/down. Avoid ↖️ ↙️ ↕️ — too niche, use words.

TODO checkbox only: [x] / [ ] are reserved for task lists (complete/incomplete items). Never use them as general pass/fail status.

Where emoji earns its tokens (chat only): status reports, drift summaries, review verdicts shown to the human. Forbidden: headers, artifacts, commits, code, shell commands.

Dropped (obscure math)

Dropped Cavekit symbols: ∀, ∃, ∴, ⊥, ∈, ∉. Use words instead: every, exists, so, never, in, not in. If a word is clearer than a symbol, use the word.

Preserve verbatim

Never compress: code blocks, paths, URLs, identifiers, numbers, error strings, SQL/regex/JSON/YAML, quoted strings.

Boundaries

  • User asks for prose explanation → switch to normal English.
  • Commit messages, PR descriptions → normal English.
  • Diff comments in code → normal English.
  • External-facing docs (RFC, pitch) → normal English.
  • Artifacts (.md files, code, shell output) → no emoji; use ASCII equivalents.
  • Chat to the human in terminal/shell → emoji OK for status/severity scanning.

Agnosticism Rules

This is the core constraint inherited from your existing AGENTS.md. The new architecture preserves and strengthens it.

Golden rule. Skills express behavior ("dispatch a reviewer", "commit a checkpoint", "check for artifacts"). The harness adapter maps behavior to tool. A skill never names a harness, model, or tool.

What skills must never contain

  • References to pi, Claude Code, Cursor, OpenCode, etc.
  • Tool names like Skill, Task, subagent, TodoWrite.
  • Harness-specific paths like ~/.claude/skills/ or ~/.pi/agent/skills/.
  • Slash-command syntax like /ck:build in the body of a skill.

What the adapter maps

Skill behaviorpi mappingOther harness
"load a skill" native skill / read harness-specific skill tool
"dispatch a subagent" subagent companion Task tool / subagent tool
"recall long-term memory" memory substrate (out of scope) skip or use native memory
"track a task" available todo tool or file harness todo tool or file

What is allowed to be harness-specific

Only Layer 4, the harness adapters. They live outside the skills themselves. We initially support two adapters:

HarnessAdapter fileKey hooks
Pi adapters/pi/extension.ts resources_discover, context, session_start, session_compact
OpenCode adapters/opencode/plugin.js config, experimental.chat.messages.transform

Both adapters do the same three things:

  1. Register the agent-skills/skills/ directory for discovery.
  2. Inject the bootstrap skill content at session start (and after compaction).
  3. Append a harness-specific tool mapping so skills' abstract behaviors resolve correctly.

Deployability guarantee

Any skill in the catalog should work in any harness that can read a SKILL.md and run Bash. Harness-specific polish (auto-trigger, subagent mapping) is added by the adapter, not baked into the skill. Adding a third harness means adding a third adapter — zero skill changes.
